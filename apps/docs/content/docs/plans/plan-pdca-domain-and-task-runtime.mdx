## Scope

Deliver the PDCA and task model in separable implementation slices under one
approved change record. This plan bootstraps task orchestration manually; once
the runtime exists, the agent derives subsequent task graphs from ready plans.

## Implementation sequence

1. **Document contracts and IA**
   - Add document classes (`living`, `decision`, `change`, `execution`).
   - Add Domain templates/catalogs for TERM, MODEL, and POLICY.
   - Add user-story lifecycle and moderated PRD review fields.
   - Update skill methodology and planning references.
2. **Schemas and validation**
   - Version `.omd` project/state contracts compatibly.
   - Add task schema and one-file-per-task loader.
   - Validate stable IDs, PLAN references, dependencies/DAG, lifecycle,
     `codeAreas`, acceptance references, and terminology config.
3. **Task generation and status**
   - Derive task proposals from PLAN sequence, scope, verification, and living
     contract references.
   - Report runnable/blocked work and next actions in JSON.
   - Add guarded task transitions.
4. **Execution orchestration**
   - Select only runnable tasks.
   - Detect overlapping active `codeAreas`.
   - Recommend remote worktrees for independent tasks and primary-agent
     subagents for locally coupled work.
   - Use shared-base completion to release dependencies.
5. **Check and Act**
   - Compare implementation/test evidence with task acceptance references.
   - Create corrective tasks or fail explicitly.
   - Complete PLAN only when required tasks and final checks pass.
6. **Handbook UI**
   - Add Domain catalogs and components.
   - Add `/docs/development/tasks` reading committed `.omd` task files.
   - Show configured labels without changing internal PLAN/TASK kinds.
7. **Distribution**
   - Refresh default templates, plugin mirrors, docs, and tests.

## Verification

- `pnpm check:planning`
- `pnpm check:ui-snapshot`
- `pnpm check:skills`
- `pnpm test`
- `pnpm typecheck`
- `pnpm build`
- Temp-repo scenarios:
  - moderated planning graph passes; unmoderated product PRD fails
  - missing/cyclic/dangling task dependencies fail
  - blocked task does not start
  - overlapping active `codeAreas` do not run concurrently
  - independent task files merge without shared-file conflict
  - Vercel-equivalent production build renders committed task snapshot

## Migration and compatibility

- Existing US documents without `stage` migrate as accepted.
- Existing SPEC documents are living contracts.
- Existing PRD/PLAN documents are change records.
- Existing `.omd` projects without tasks remain valid until orchestration is
  requested.
- Internal IDs remain PRD/US/SPEC/ADR/PLAN/TASK; terminology config changes
  presentation only.

## Risks and rollback

- Overly strict moderation can block maintenance: enforce it only for product
  PRDs.
- Declared `codeAreas` may miss semantic conflicts: present them as a minimum
  exclusion mechanism, not proof of independence.
- Task state may become noisy in Git: one file per task and bounded transitions
  limit contention.
- Roll back individual slices while preserving living document IDs and task
  schema compatibility.

## Completion criteria

- Natural-language planning and implementation requests follow the PDCA gates.
- Domain language and policies are first-class living documents.
- Agent-generated tasks are validated, dependency-aware, and base-approved.
- Check/Act uses concrete contract/test evidence.
- The static handbook task board matches the deployed Git commit.
