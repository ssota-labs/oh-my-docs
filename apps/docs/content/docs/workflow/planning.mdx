## 1. Classify the change

| Change | Required artifacts |
|---|---|
| Product behavior | PRD, story, specification, implementation plan |
| Bug fix | Existing PRD and specification, implementation plan |
| Maintenance | Implementation plan; specification if a contract changes |
| Documentation only | No prior plan |

Reuse an existing artifact when it already explains the change. Do not create
new IDs for the same intent.

## 2. Write in dependency order

Write and review:

`vision → PRD → story → specification/ADR → implementation plan → code`

PRDs and stories use language that product reviewers can evaluate without
knowing the code. Specifications define observable behavior. ADRs preserve a
choice and rejected alternatives. Plans authorize a bounded implementation.

## 3. Complete the graph

Use templates under the docs app `templates/` directory, or create drafts with
the skill runtime:

```bash
node <skill>/scripts/omd.mjs new prd|story|spec|plan|adr --title "…" --yes
```

Give every artifact a stable, kind-prefixed ID, link it through frontmatter, and
register its filename in the sibling `meta.json` (`new` does this).

Set an approved PRD to `active`, a reviewed specification to `accepted`, and an
approved plan to `ready`. Plans that authorize code must declare non-empty
`codeAreas`.

## 4. Review planning separately

Merge or otherwise approve the planning change before implementation starts.
Do not add a missing ready plan and its code in one change.

## 5. Validate

```bash
pnpm check:planning
pnpm check:docs-first   # CI on pull requests; skips locally without BASE_SHA
node <skill>/scripts/omd.mjs check --json
```
