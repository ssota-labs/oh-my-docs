## Scope

Defines the persistent Git snapshot for task identity, dependencies, execution
scope, lifecycle, and acceptance references.

## Interface

```text
.omd/
  project.json
  state.json
  tasks/
    TASK-<stable-id>.json
```

One task per file is required to minimize merge conflicts between worktrees.

## Behavior

An agent generates tasks from a base-branch PLAN in `ready|active`. Task files
are proposed and committed before or with execution-state transitions. They
remain inspectable after completion.

## Data and invariants

Required task fields:

```json
{
  "id": "TASK-task-runtime-001",
  "planId": "PLAN-pdca-task-runtime",
  "title": "Implement task state validation",
  "status": "ready",
  "dependsOn": [],
  "codeAreas": ["skills/oh-my-doc/runtime"],
  "acceptance": ["POLICY-task-runnable", "SPEC-task-system-model"]
}
```

- ID prefix is `TASK-` and immutable after publication.
- One task belongs to exactly one PLAN.
- Status values are `draft`, `ready`, `blocked`, `active`, `done`, `failed`,
  `cancelled`.
- Dependencies reference task IDs in the same project and form a DAG.
- `codeAreas` is non-empty before `ready`.
- Acceptance references use stable US, POLICY, or SPEC IDs.
- Internal kinds remain `PLAN` and `TASK`; configured labels do not change IDs.

## Errors

- Duplicate/malformed IDs, missing PLANs, dangling dependencies, cycles, and
  invalid status values fail `check`.
- Ready tasks with empty `codeAreas` or acceptance references fail `check`.

## Compatibility

Task state is additive to the current `.omd` contract. Projects without tasks
remain valid until they request implementation orchestration.

## Acceptance criteria

- JSON schema validates task files independently.
- Updating one task does not rewrite unrelated task files.
- A deterministic task graph can be reconstructed from a commit.
- Configured “Plan”/“Sprint” display labels do not change storage semantics.
