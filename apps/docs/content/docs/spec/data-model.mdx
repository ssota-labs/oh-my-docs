## Scope

Defines the durable data shape of handbook planning artifacts and the committed
Git snapshot for agent execution tasks. Routes, orchestration behavior, and
sidebar IA are covered by [System model](/docs/spec/system-model). Runtime
actions are covered by [CLI](/docs/spec/cli).

Absorbs the former standalone task data contract (`SPEC-004`).

## Handbook entities

| Kind | ID prefix | Catalog path | Lifecycle field |
|---|---|---|---|
| Product requirements | `PRD-` | `planning/prds` | `status` |
| User stories | `US-` | `planning/stories` | (owned by linked PRD/plan) |
| Specifications | `SPEC-` | `spec/<typed-page>` | `stage` |
| Architecture decisions | `ADR-` | `adr` | `stage` |
| Implementation plans | `PLAN-` | `plans` | `stage` |
| Execution tasks | `TASK-` | `.omd/tasks/` (not a sidebar catalog) | `status` |

## Frontmatter invariants

- Every catalog document has a stable `id` matching its kind prefix.
- Plans declare `changeType`, references (`prd` / `specs` / `stories`),
  lifecycle `stage`, and non-empty `codeAreas` when authorizing implementation.
- Sibling `meta.json` files explicitly register each catalog entry under `pages`.
- Spec documents are living typed pages; they keep `SPEC-*` IDs even when the
  filename is a boundary name (`data-model`, `system-model`, `cli`).

## Task snapshot (`.omd`)

```text
.omd/
  project.json
  state.json
  tasks/
    TASK-<stable-id>.json
```

One task per file minimizes merge conflicts between worktrees.

Required task fields:

```json
{
  "id": "TASK-task-runtime-001",
  "planId": "PLAN-pdca-task-runtime",
  "title": "Implement task state validation",
  "status": "ready",
  "dependsOn": [],
  "codeAreas": ["skills/oh-my-doc/runtime"],
  "acceptance": ["POLICY-task-runnable", "SPEC-handbook-information-architecture"]
}
```

- ID prefix is `TASK-` and immutable after publication.
- One task belongs to exactly one PLAN.
- Status values are `draft`, `ready`, `blocked`, `active`, `done`, `failed`,
  `cancelled`.
- Dependencies reference task IDs in the same project and form a DAG.
- `codeAreas` is non-empty before `ready`.
- Acceptance references use stable US, POLICY, or SPEC IDs.
- Internal kinds remain `PLAN` and `TASK`; configured labels do not change IDs.

An agent generates tasks from a base-branch PLAN in `ready|active`. Task files
are proposed and committed before or with execution-state transitions.

## Validation

The planning check fails on duplicate or malformed IDs, missing references,
invalid ready-state dependencies, missing code areas, or unregistered catalog
documents. Task checks fail on duplicate/malformed task IDs, missing PLANs,
dangling or cyclic dependencies, empty `codeAreas` on ready tasks, or invalid
status values.

## Compatibility

Brownfield handbooks that still use `development/plans` and `development/adr`
must migrate paths with the IA flatten. IDs do not change when files move.
Task state is additive to the current `.omd` contract; projects without tasks
remain valid until they request implementation orchestration.
