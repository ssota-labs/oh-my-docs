## Scope

Covers the skill-era runtime actions that install, inspect, and refresh the
planning UI vocabulary snapshot. Does not cover Fumadocs shell package versions
on npm.

## Interface

| Action | Role |
|---|---|
| `inspect --json` | Report docs/UI/`.omd` presence and UI snapshot health hints |
| `check --json` | Fail when `.omd` is missing, vocabulary contract is incomplete, or UI snapshot is missing/stale vs skill template |
| `adopt --yes` | Create missing UI files from the skill template; skip identical files |
| `adopt --yes --force` | Also overwrite UI files that differ from the skill template |
| `sync --yes` | Refresh managed IA/`AGENTS.md`/`CLAUDE.md` markers only (not UI files) |

Runtime entrypoint: `node <skill>/scripts/omd.mjs <action>`.

## Behavior

1. **Missing UI package or files** — `doctor`/`inspect` note a missing UI path;
   `check` includes a problem; `adopt` creates files from
   `<skill>/templates/default/packages/ui` into the configured `ui` path
   (default `packages/ui`).
2. **Identical snapshot** — `adopt` skips with reason `UI snapshot up to date`.
3. **Divergent files** — without `--force`, `adopt` records a conflict/skip;
   `check` reports staleness so agents can ask before forcing.
4. **Forced refresh** — `adopt --force` updates divergent skill-template UI
   files and refreshes `.omd` state digests related to the snapshot.
5. **Skill vs project** — installing/updating the skill alone does not mutate
   the project; agents must run `adopt`/`check` against the project.

## Data and invariants

- `.omd/project.json` declares `ui.distribution: skill-template` and the
  vocabulary list.
- `.omd/state.json` records `uiSnapshot.vocabularyDigest` (and related file
  digests when written by adopt).
- Project UI lives under the configured `paths.ui`; the skill template remains
  the source of truth for refresh.

## Errors

| Condition | Signal |
|---|---|
| No `.omd/project.json` | `check` problem: run adopt first |
| UI path missing | doctor/inspect missing note; check problem |
| Vocabulary name missing from contract | `check` problem naming the component |
| Template file differs and not forced | adopt conflict/skip; check stale problem |

## Compatibility

- Brownfield projects that already customized `packages/ui` must use `--force`
  knowingly; adopt never silently overwrites divergent files without it.
- `sync` must not rewrite UI vocabulary files.

## Acceptance criteria

- `check --json` returns `ok: false` with an explicit UI missing/stale problem
  when the project snapshot does not match the skill template for managed UI
  files (identical content is not stale).
- `adopt --yes` restores missing files without `--force`.
- `adopt --yes --force` makes the project UI match the skill template again.
- Machine-readable JSON includes enough detail for an agent to choose adopt vs
  adopt `--force`.
