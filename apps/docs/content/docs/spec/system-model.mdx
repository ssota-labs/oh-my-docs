## Scope

Observable navigation, rendering, and orchestration contracts for the handbook
app, default skill template IA, and approved task execution. Runtime entrypoints
are covered by [CLI](/docs/spec/cli). Persistent shapes live in
[Data model](/docs/spec/data-model).

Absorbs the former standalone contracts for PDCA workflow (`SPEC-003`), task
execution (`SPEC-005`), and task board UI (`SPEC-006`).

## Routes

- `/` redirects to `/docs` (handbook entry).
- `/docs` and `/docs/<slug>` render handbook pages.
- `/docs/<slug>.md` returns the corresponding processed Markdown with
  `text/markdown; charset=utf-8`.
- `/docs/plans/tasks` (target route; formerly drafted as `/docs/development/tasks`)
  renders the read-only task board from the checked-out `.omd` snapshot.
- Unknown page slugs return not found.

## Default information architecture

Top-level sidebar order:

`Home → Vision → Start here → Domain → Workflow → Planning → Plans → ADR → Spec`

| Section | Sidebar behavior |
|---|---|
| Domain | Living glossary, models, and policies |
| Planning | Folder with index-only catalogs: PRDs, Stories |
| Plans | Index-only catalog at the top level |
| ADR | Index-only catalog at the top level |
| Spec | Boundary entries; Data model and System model are index-only folders |

There is no intermediate Development section and no default Agent section.

## Index-only catalogs

Configured catalog folders keep detail pages registered in `meta.json` for
validation, but the page tree exposes only the catalog index. Opening the
index shows a table of documents (ID, status when present, title). Detail pages
provide back-to-index and sibling previous/next links.

Default catalog prefixes:

- `planning/prds` → `/docs/planning/prds`
- `planning/stories` → `/docs/planning/stories`
- `plans` → `/docs/plans`
- `adr` → `/docs/adr`
- `spec/data-model` → `/docs/spec/data-model` (after index-only boundary catalogs land)
- `spec/system-model` → `/docs/spec/system-model` (after index-only boundary catalogs land)

## Spec typed boundaries

Spec is not a flat `SPEC-*` catalog. The sidebar lists a stable set of
boundaries:

- [Data model](/docs/spec/data-model)
- [System model](/docs/spec/system-model)
- [CLI](/docs/spec/cli)

**Data model** and **System model** are index-only folders (same mechanism as
PRDs): the sidebar shows the boundary entry; detail pages live under the folder
and appear in the boundary index table, not in the sidebar. CLI may stay a
single page until it needs children.

Initiative-specific behavior belongs under these boundaries (as hidden child
pages or sections), not as an unbounded top-level Spec list.

## PDCA planning lifecycle

User-facing phases:

`Discover → Plan → Design → Approve → Do → Check → Act`

| Phase | Primary artifacts |
|---|---|
| Discover / Plan | Candidate US, moderated PRD |
| Design | Domain, Spec boundaries, ADR, PLAN |
| Approve | Ready PLAN (and task graph when present) on shared base |
| Do | Agent-generated `.omd` tasks and implementation |
| Check / Act | Evidence vs acceptance; corrective tasks or PLAN completion |

## Task orchestration

Natural-language operations:

- “Start implementation” — inspect approval and task state, generate tasks if
  absent, and start runnable work.
- “Show status” — report phase, task states, blockers, and next actions.
- “Check implementation” — compare completed work with acceptance references.

Behavior:

1. Resolve a PLAN from the base branch with stage `ready|active`.
2. If no task graph exists, derive a proposal from implementation sequence,
   `codeAreas`, verification, and referenced living contracts.
3. Validate the graph and commit/approve its snapshot.
4. Select only tasks satisfying `POLICY-task-runnable`.
5. Partition independent tasks:
   - remote worktrees for non-overlapping, independently mergeable work;
   - one primary agent plus subagents for tightly coupled local work.
6. Transition selected tasks to `active`.
7. Run implementation and focused verification.
8. Check implementation against US/POLICY/SPEC acceptance references.
9. Mark tasks `done` or `failed`; release blocked dependents only after the
   predecessor is `done` on the shared base.

Invariants:

- No task runner modifies another task's status file.
- Overlapping active `codeAreas` are mutually exclusive.
- Branch-local `done` does not release a dependent until merged into the shared
  base.
- The orchestrator never auto-merges task branches.
- PLAN completion requires all non-cancelled tasks done and final Check passing.

## Task board view

Read-only handbook rendering of committed task state (former `SPEC-006`).

- Source: `.omd/tasks/*.json` from the checked-out commit
- Presentation: PLAN grouping, status, dependencies, blockers, `codeAreas`, and
  acceptance references
- Local docs reflect the local filesystem after rebuild/reload; Vercel reflects
  the deployed commit
- Empty projects show an explicit “no approved tasks” state
- Invalid snapshots fail validation rather than presenting partial truth
- The board never writes `.omd`
- Display labels may map PLAN to “Plan”/“Sprint”; stored IDs remain unchanged

This is a handbook **view** of data already defined in
[Data model](/docs/spec/data-model), not a separate product surface that needs
its own Spec sidebar entry.

## Source

Content lives under the configured docs content root (this repo:
`apps/docs/content/docs`). Fumadocs validates frontmatter and generates the
page tree. Skill `adopt` / `sync` keep template `meta.json` and catalog wiring
aligned with `.omd` IA defaults.

## Acceptance criteria

- Catalog detail pages do not appear as sidebar children.
- Catalog indexes render tables, not long unstructured link stacks alone.
- ADR and Plans are reachable as second-level sidebar items.
- Spec sidebar shows only the boundary entries; Data model and System model
  children stay index-only like PRDs.
- Runnable task selection is deterministic for the same Git commit.
- A committed task appears in local and deployed handbook builds; uncommitted
  state does not.
