As a contributor, I want the agent to start only approved and dependency-ready
work so that parallel implementation does not bypass product review or create
avoidable merge conflicts.

## Context

PLAN remains the reviewed change record. After it is ready on the base branch,
the agent derives execution tasks into `.omd`. A task is not a renamed PLAN or
sprint: it is the stable internal execution unit regardless of configured UI
labels.

## Acceptance criteria

- Tasks are generated by the agent after the linked PLAN is ready or active on
  the base branch.
- A runnable task has non-empty `codeAreas`, status `ready`, all dependencies
  `done`, and no conflict with another active task.
- Dependency cycles fail validation.
- Independent tasks may be assigned to separate worktrees.
- Closely related tasks may stay with one primary agent using subagents.
- Starting a task changes only that task's execution-state file.

## Out of scope

- A remote real-time scheduler or shared database.
- Automatic merging of task branches.
