## Context

The docs shell needs Fumadocs packages from npm. Planning vocabulary components
(`DocKind`, `Decision`, catalog helpers, …) are Oh My Docs-specific and must
reach consumer repos without a public CLI or a second package registry.

## Decision

Ship planning UI under `skills/oh-my-doc/templates/default/packages/ui` and copy
it into the consumer project's `packages/ui` (or configured `paths.ui`) during
`adopt`. Refresh the same way. Keep Fumadocs shell dependencies on npm.

## Consequences

- No shadcn-style registry or `npx` component pull for planning vocabulary.
- Skill updates do not mutate projects until `adopt` runs again.
- Divergent local edits require explicit `adopt --force`.
- Dogfood `packages/ui` and the skill template must stay in sync in this repo.

## Rejected alternatives

- **Publish `@oh-my-docs/ui` publicly** — adds release surface and versioning
  churn before multiple external consumers exist.
- **shadcn registry** — couples planning vocabulary to a registry protocol and
  implies pick-and-choose installs; Oh My Docs needs a coherent snapshot.
- **Import live from the skill folder at runtime** — host skill paths vary and
  must not become a build dependency of the docs app.
