## Context

The handbook must serve people through a navigable site and agents through
plain Markdown without maintaining two content sets.

## Decision

Use Fumadocs MDX as the content and schema layer inside a minimal Next.js app.
Enable processed Markdown during compilation and expose it through rewritten
`.md` routes. Keep required components local until shared UI has a demonstrated
cross-application use.

## Consequences

- A single source produces both representations.
- Frontmatter is validated while the site builds.
- Generated `.source` output remains untracked.
- The docs app owns a small amount of local presentation code.

## Rejected alternatives

- A separate generated Markdown repository risks drift and adds publication work.
- Parsing raw files directly in routes bypasses Fumadocs processing semantics.
- A Phase 1 shared UI package would create an abstraction before a second
  consumer exists.
