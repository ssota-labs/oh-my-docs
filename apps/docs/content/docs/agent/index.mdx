## Source of truth

Use this handbook for product intent, observable contracts, decisions, and
reviewed implementation plans. Each browser page has a processed Markdown twin at
the same URL with `.md` appended.

`AGENTS.md` is the canonical instruction file for coding agents. `CLAUDE.md`
only bridges to it.

## Install the skill

Users do not run a public Oh My Docs CLI. Install the skill, then drive the
bundled runtime:

```bash
npx skills add ssota-labs/oh-my-docs --skill oh-my-doc -y
```

### Discovery paths

| Host | Project | User |

| Cursor | `.cursor/skills/oh-my-doc` | `~/.cursor/skills/oh-my-doc` |

In this repository the canonical skill source is `skills/oh-my-doc`. Optional
host plugin wrappers under `plugins/{cursor,codex,claude-code}` mirror it for
marketplace discovery.

## Adopt and check a project

```bash
node <skill>/scripts/omd.mjs inspect --json
node <skill>/scripts/omd.mjs adopt --yes --json
node <skill>/scripts/omd.mjs check --json
```

| Action | Purpose |
|---|---|
| `inspect` | Classify greenfield vs brownfield; report docs/UI/`.omd` state |
| `adopt` | Scaffold or import handbook, write `.omd/`, copy UI vocabulary into `packages/ui` |
| `adopt --force` | Also overwrite UI files that diverge from the skill template |
| `check` | Validate planning graph, `.omd` contract, and UI snapshot health |
| `sync` | Refresh managed IA pages and `AGENTS.md` / `CLAUDE.md` markers (not UI) |
| `new` | Create `prd` / `story` / `spec` / `plan` / `adr` drafts |

Greenfield scaffolds land under `docs/content/docs`. Brownfield keeps an
existing `docs/` or `apps/docs` tree in place.

Do not hand-edit managed `<!-- oh-my-docs:* -->` marker blocks; run `sync` or
`adopt`.

### Skill or UI updates

Updating the skill does not rewrite the project. When `check` or `inspect`
reports missing or stale UI vocabulary, refresh from the skill template:

```bash
node <skill>/scripts/omd.mjs adopt --yes --json
# if files diverge intentionally or look stale:
node <skill>/scripts/omd.mjs adopt --yes --force --json
```

Fumadocs shell packages (`fumadocs-ui`, `fumadocs-core`, `fumadocs-mdx`) stay on
npm. Planning vocabulary is a skill-template snapshot, not a component registry.

## Before changing code

1. Read the repository `AGENTS.md`.
2. Follow the [planning workflow](/docs/workflow/planning).
3. Confirm the change is covered by a ready or active
   [implementation plan](/docs/plans).
4. Use the [development contract](/docs/workflow/development) for validation and
   completion requirements.

Do not invent product behavior from code alone, and do not create an
implementation together with the plan that first authorizes it.
